## Overall Assessment

The site is in good shape. The security posture is noticeably above average for a marketing site, the terminal aesthetic is cohesive, and reduced-motion support is one of the better implementations I've seen. The issues below are mostly polish, a couple of real bugs, and a few accessibility gaps worth closing.

## Security

**Strengths**

- Nonce-based CSP with `strict-dynamic` â€” correct and strong pattern for Next.js
- `frame-ancestors 'none'` combined with `X-Frame-Options: DENY` (belt and suspenders, fine)
- `object-src 'none'`, `form-action 'self'`, `upgrade-insecure-requests` all present
- HSTS with `includeSubDomains; preload` â€” good
- Permissions-Policy, COOP, X-Content-Type-Options, Referrer-Policy all set
- Dev vs prod CSP branching is correctly implemented (`unsafe-eval` only in dev)
- CloudFront media host whitelisted in `img-src` and `media-src`, not broader

**Issues**

1. **No CSP violation reporting.** There's no `report-uri` or `report-to` directive and no `Content-Security-Policy-Report-Only` header in staging. Violations in production are invisible. Add a reporting endpoint (even a free one like Report URI) to catch breakage early.

2. **`blob:` in `script-src` is permissive.** Allowing blob-URI script execution is a meaningful CSP relaxation. Verify this is actually required by Next.js internals in production; if it's only needed for dev tooling, scope it to the dev branch.

3. **`font-src data:`** allows inline data-URI fonts. Low risk for a static site, but worth auditing whether it's actually needed now that `@fontsource/ibm-plex-mono` serves real font files.

4. **`X-Frame-Options: DENY` is redundant** once `frame-ancestors 'none'` is in the CSP. Not harmful, but creates maintenance drift.

5. **Missing `Cross-Origin-Resource-Policy` header.** Not critical, but given COEP and COOP are already set, adding `CORP: same-origin` or `same-site` would complete the isolation triplet.

## Code Quality

**Strengths**

- TypeScript used consistently across pages and components
- Clean App Router patterns â€” async server components, proper `generateStaticParams` for SSG
- Good data/component/page separation
- `serviceDefinitions` as a single source of truth for service slugs and visuals
- `useMemo` used appropriately to avoid translation object churn

**Issues**

1. **`useScrollPosition.js` is plain JavaScript.** Everything else is `.ts`/`.tsx`. This hook has no types and lives in `src/components/hooks/` rather than a dedicated `src/hooks/`. More importantly, it appears to be unused â€” no import found in any component. It should either be typed and moved, or deleted.

2. **`useScrollPosition` has a missing dependency array bug.** The `useEffect` inside has no `[]` dependency array, so it re-subscribes the scroll listener on every render. This would cause a performance problem if the hook were used.

3. **`t.raw()` casts are unvalidated.** `t.raw(\`...\`) as string[]` and `as { title: string; text: string }[]` are runtime casts with no shape validation. If a translation key is missing or structured incorrectly, this fails silently or throws a confusing error. Consider a Zod parse or at minimum a runtime check in development.

4. **Mixed `NextLink` / `Link` in `services/[slug]/page.tsx`.** The file imports both `next/link` (`NextLink`) and the locale-aware `@/i18n/navigation` (`Link`) and uses both. The `mailto:` CTA uses `NextLink` â€” fine, since it's an external href. But the inconsistency is easy to trip over when adding links later. Add a comment or consolidate.

5. **`page.tsx.off` sitting in the source tree.** The disabled contact page (`src/app/pages/contact/page.tsx.off`) is not being served but is checked in. Either remove it or move it to a `_drafts/` folder so it's clearly not dead code.

6. **Color token type mismatch in `globals.css`.** In dark mode, `--color-bkg2` is set to `0deg 0% 0%` (HSL-style), but the `@theme` block wraps it with `rgb()` â†’ `rgb(0deg 0% 0%)` which is invalid CSS. If `bkg2` is ever applied, it will silently produce no color. Verify all token/wrapper combinations are consistent.

7. **`--font-sans` and `--font-mono` are identical.** Both point to IBM Plex Mono. The `font-sans` alias is misleading. Since this is intentional (monospace-only design), either remove `font-sans` or rename it to make the intent explicit.

## Design & UX

**Strengths**

- Terminal aesthetic is cohesive and well-executed throughout
- `prefers-reduced-motion` is handled correctly in both CSS and JS â€” the JS side skips all timers and shows content immediately, and the CSS side disables animations. This is the right pattern.
- Responsive breakpoints (sm/md/lg) are consistent and don't have obvious collapse issues
- Semantic HTML is generally good â€” `article`, `section`, `h1/h2/h3` hierarchy is correct
- i18n (EN/DE) is cleanly integrated

**Issues**

1. **Mobile menu doesn't close on link click.** When a nav link is tapped on mobile, `isOpen` stays `true` and the menu remains visible over the new page. Add an `onClick={() => setIsOpen(false)}` to each mobile menu `<li>` link, or use a `useEffect` that closes on route change.

2. **Hamburger button is missing an accessible label.** The mobile menu `<button>` has `aria-expanded` but no `aria-label` or `aria-controls`. Screen readers will announce it as an unlabeled button. Add `aria-label={isOpen ? "Close menu" : "Open menu"}` (localised).

3. **Logo `alt` text is generic.** `alt="Logo"` gives screen readers no information. Change to `alt="Niwo Systems"` or `alt="Niwo Systems â€” home"`.

4. **Background video is not marked as decorative.** The `<video>` element has no `aria-hidden="true"` or `role="presentation"`. Assistive technologies may announce it. Since it's a pure visual background, add `aria-hidden="true"`.

5. **Navbar services link bypasses locale routing.** `href="/#what-i-do"` is a root-relative path. When viewed at `/en/...` or `/de/...`, this drops the locale prefix and triggers a redirect. Use `href="/"` with a hash via the i18n `Link`, or programmatically build the locale-prefixed hash link.

6. **Light-mode color tokens appear vestigial.** The `globals.css` defines extensive light-mode tokens (`--color-bkg1`, `--color-content1`, etc.) but the site is fully dark â€” `body` is hardcoded `bg-black`. If light mode is planned, these are fine. If not, they're noise that makes maintenance harder.

## Performance

**Strengths**

- `generateStaticParams` on service pages â€” all three service routes are statically generated at build time
- `preload="metadata"` on the background video â€” avoids unnecessary data load
- `viewport={{ once: true }}` on Framer Motion â€” animations don't re-fire on scroll-up
- Next.js `<Image>` used for the logo with explicit `width`/`height`
- Turbopack in dev

**Issues**

1. **No `poster` attribute on the background video.** Before the video loads (or if it fails), users see a blank/black area. A static poster image (a frame from the video, or a dark gradient PNG) would improve perceived load time on slow connections and eliminate any flash.

2. **Mass `setTimeout` usage for typing animation.** The headline effect schedules one `setTimeout` per character â€” roughly 50+ timers for the headline alone, plus hundreds more for the terminal block. These all accumulate in the timer queue. This works fine in practice but a single `setInterval`-based approach would be cleaner and easier to cancel. Not a measured performance issue, just a code smell.

3. **`terminalSignature` is a spurious `useEffect` dependency.** `terminalLines.join("\n")` is computed and used only as a dependency, not in the effect body. This means the effect re-runs if line content changes (which shouldn't happen mid-session), and the intent isn't obvious to a reader. The direct `terminalLines` dependency already covers this â€” `terminalSignature` can be removed from the dep arrays.

4. **Framer Motion for simple card hover.** Framer Motion (~40kB gzipped) is used for `ServiceCard` entry animations and hover lift. The same effect is achievable with CSS `@keyframes` and `transition`. For a marketing site this bundle cost is acceptable, but worth noting if Core Web Vitals become a concern.

5. **Double `getTranslations` call in `ServicePage`.** Both `generateMetadata` and the page component call `getTranslations({ locale })` independently. In a server component, Next.js will cache this per-request, so it's not a correctness issue â€” but it's worth confirming caching behavior across Next.js versions.

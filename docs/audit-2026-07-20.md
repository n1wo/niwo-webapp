# niwo-webapp Audit — 2026-07-20 (Web Discoverability & Services Go-Live)

Scope: the WEB-5 go-live update merged in PR #52 — crawler policy, entity identity, four service landing pages, breadcrumbs/internal linking, metadata system, CI discoverability gate.
Stack: Next.js 16 / React 19 / TypeScript / Tailwind CSS v4 / next-intl, deployed via AWS Amplify behind Cloudflare.
Audit basis: commit `989ba48` on `main`, local dev server, production responses from `https://www.niwosystems.com`, and the WEB-44…WEB-47 acceptance criteria.

---

## Overall Assessment

The update is in good shape and shipped verified: all 40 sitemap routes pass the new CI discoverability gate (unique metadata, canonical/hreflang parity, one h1, valid JSON-LD, orphan detection), the crawler policy is explicit and production-tested for five agents, and content guardrails from the portfolio decision are respected (no prices, no overclaims, `/insights` gated). The issues below are two real fixes (robots blocking render-critical CSS, missing favicon fallback), several consistency improvements, and cleanups. Nothing found that argues against the pages staying live.

## Discoverability & SEO

**Strengths**

- Explicit per-agent robots policy (Googlebot/Bingbot/OAI-SearchBot allow, GPTBot training opt-out, ChatGPT-User allow) — documented in `docs/crawler-policy.md` and verified in production after the Cloudflare legacy "Block AI bots" rule was disabled
- Organization, Service, TechArticle, and BreadcrumbList JSON-LD all validate and match visible content
- `x-default` hreflang added; canonicals, localized OG/Twitter images and alt text consistent across EN/DE
- Sitemap generated from a single route source (`siteRoutes.ts`) with per-content `lastModified` for topics
- Identity statement server-rendered on both homepages and regression-checked in CI (`data-niwo-identity`)
- Legacy `/services/<topic>` URLs 308-redirect to `/topics/<topic>` — no duplicate-content surface

**Issues**

1. **`Disallow: /_next/` blocks render-critical CSS/JS for all crawlers.** Pages reference stylesheets under `/_next/static/…` (verified in production HTML), and robots.txt disallows `/_next/` for every agent. Essential content is server-rendered, so indexing works, but Google explicitly recommends not blocking resources needed for rendering — this can affect mobile-friendliness and page-experience evaluation. Change the rule to allow `/_next/static/` and `/_next/image` while keeping the rest of `/_next/` blocked.

2. **Service routes carry a pre-launch `lastModified`.** `STATIC_CONTENT_UPDATED = "2026-07-12"` predates the go-live (2026-07-20), so the sitemap claims the service pages were last modified before they existed publicly. Bump the constant on content-bearing releases, or track a per-service date the way topics do.

3. **Cloudflare prepends a managed content-signals block to robots.txt.** The app policy survives intact below it, and the duplicate `GPTBot Disallow` is harmless, but the served file no longer equals the app-generated file. `check-crawlers.mjs` covers this; keep `docs/crawler-policy.md` as the record of the two-layer interplay whenever Cloudflare settings change.

4. **Service schema is minimal.** The `Service` JSON-LD has name/description/url/provider/areaServed/channel but no `serviceType` or `inLanguage`. Optional enrichment; keep it truthful — no `offers`/pricing until prices are approved.

5. **No FAQ section on service pages.** The WEB-46 content mask lists FAQ as a reusable section; the shipped pages cover who/what/scope/process/limits/next-step without one. Acceptable as a conscious deferral — add FAQs only when they answer real recurring questions, not for keyword volume.

## Security

**Strengths**

- Full security-header set (nonce-based CSP, HSTS, COOP/CORP, Permissions-Policy, frame/content-type controls) preserved through the middleware→`proxy.ts` migration — CI asserts CSP plus seven companion headers
- JSON-LD injected as `application/ld+json` (non-executable; not a CSP script-src concern), content is build-time constants, no user input
- Contact page instructs users not to send credentials, private logs, patient data, or incident evidence by email
- Crawler policy avoids UA-based trust: `docs/crawler-policy.md` explicitly requires provider IP/rDNS verification for any WAF/CDN exception
- `/insights` hard-gated (empty data → 404, absent from sitemap and nav) until author-reviewed content exists

**Issues**

1. **Proxy matcher exclusions skip security headers on excluded paths.** `/assets/…`, `favicon.ico`, `sitemap.xml`, `robots.txt`, `manifest.webmanifest`, and `/.well-known/security.txt` bypass `proxy.ts` and are served without the header set. Low risk for static XML/images, but `X-Content-Type-Options: nosniff` on assets would be preferable if the hosting layer can add it.

2. **Two independent bot-control layers now exist (app robots.txt and Cloudflare AI policies).** Today they agree; nothing enforces that they keep agreeing. The production `check:crawlers` run is the drift detector — run it after any Cloudflare change (see Measurement doc / WEB-50).

## Code Quality

**Strengths**

- `services.ts` is a clean single source of truth: slugs, topic links, and full EN/DE content in one typed structure feeding pages, sitemap, navbar, and related-services blocks
- `siteRoutes.ts` centralizes the public-route inventory for sitemap, robots, and checks — new routes cannot silently miss the sitemap
- Shared `Breadcrumbs` component emits visible nav and matching `BreadcrumbList` JSON-LD from the same items array (schema/UI cannot drift)
- CI gate (`check:discoverability`) is strict and cheap: metadata uniqueness, canonical/og:url consistency, hreflang parity, h1 count, JSON-LD parseability, orphan detection, HTML size budget, alt-attribute presence
- Topic pages now derive related services from `topicSlugs` — one relationship, rendered both directions

**Issues**

1. **Bilingual copy now lives in three patterns.** `messages/{en,de}.json` (homepage/nav), `data/services.ts` (services), and page-local `copy` consts (services index, contact, phishing-lab). All are typed, but future translators must know three places. Consider consolidating page-local consts into the messages files or documenting the split rule (data-driven pages keep copy in `data/`).

2. **`locale === "de" ? "de" : "en"` fallback is duplicated across ~6 files.** A `resolveAppLocale(locale)` helper in `i18n/` would remove the repetition and give one place to extend when a third locale arrives.

3. **`check-routes.mjs` parses HTML with regexes tied to React's attribute casing** (`hrefLang="…"`). Works today and CI would fail loudly, but a Next.js rendering change could produce confusing false negatives. A tolerant regex (`hreflang` case-insensitive) or a tiny HTML parser would be more robust.

4. **`check-crawlers.mjs` exit code is easy to mask.** Piping through `tail` in ad-hoc runs hides the non-zero exit (this happened during go-live verification and briefly hid the ChatGPT-User 403). Print an explicit final `PASS`/`FAIL` line so truncated output still shows the verdict.

5. **Root redirect layout hardcodes `lang="en"`.** `(root)/layout.tsx` only ever renders during the `/`→`/en` redirect, so impact is nil; a comment stating that would prevent a future "bug" report.

## Design & UX

**Strengths**

- Service pages answer who/what/included/excluded/process/deliverables/prerequisites/availability in scannable, parallel EN/DE structure — the "not included" sections are unusually honest and align with the portfolio guardrails
- Breadcrumbs are visible, localized (`Brotkrumen`/`Breadcrumb` aria-labels), and keyboard-reachable
- Descriptive anchors throughout ("View Microsoft 365 review", topic names) — no bare "learn more"
- Navbar gained a real Services entry on desktop and mobile; contact links now route to the contact page instead of bare `mailto:`
- Contact page separates business vs. security contact and states data-safety expectations up front

**Issues**

1. **`/favicon.ico` returns 404 in production and dev.** Only the SVG favicon is declared; browsers, feed readers, and some crawlers still request `/favicon.ico` unconditionally. Add a real `favicon.ico` in `app/` (Next serves it automatically) or a redirect to the SVG.

2. **Service-page CTA bypasses the contact page.** The CTA is a `mailto:` with prefilled subject, while the navbar deliberately moved contact to `/pages/contact`, which carries the "don't email sensitive evidence" guidance. Someone on the incident page — the audience most likely to paste sensitive details into a first mail — never sees that warning. Either link the CTA to `/pages/contact`, or repeat the one-line safety note next to the mailto button.

3. **Topic pages show breadcrumb and "back to topics" link directly adjacent.** Redundant affordance; harmless, but removing the old back-link would tighten the header.

4. **Manifest has a single SVG icon.** No PNG/maskable sizes, so installed-app icons on some Android launchers fall back poorly. Two PNGs (192/512, one maskable) would complete it. `theme_color`/`background_color` match the site.

5. **Related-topic links on service pages render raw slugs** (`incident response →` from `slug.replaceAll("-", " ")`) instead of the topic's real localized title — on DE pages the anchors stay English. Use the topic definition's title for the label.

## Performance

**Strengths**

- All service and topic routes are statically generated (`generateStaticParams`); sitemap/robots/manifest are build-time
- OG/Twitter images are generated per locale (not per page), cached by the framework — two images serve all 40 routes
- CI enforces a 700 KB HTML budget per route
- No new client-side JS on service pages — they are fully server-rendered list markup

**Issues**

1. **`check:discoverability` fetches all 40 routes sequentially.** CI time grows linearly with content; batching fetches (e.g., 5-way `Promise.all`) would keep the gate fast as topics/insights grow. Not urgent at 40 routes.

2. **Homepage animation/CWV impact remains unaudited.** The identity statement guarantees crawlable content, but WEB-49 (animation crawlability, Core Web Vitals) is still open — this audit did not measure LCP/INP. Treat WEB-49 as the follow-up, ideally with field data once Search Console has impressions.
